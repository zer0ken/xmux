xmux — 이상적 구조 리팩토링

마스터 설계 스펙 · 2026-07-02 · 브랜치 refactor/ideal-structure (워크트리 .claude/worktrees/refactor-ideal, 베이스 0818679) · 상태: 설계 — 스펙 검토 대기

이 스펙은 코드 품질 rubric(SOLID, 응집도, 결합도, DRY/WET, KISS/YAGNI, 관심사 분리, 데메테르의 법칙, CQS, 명시성, Fail-Fast/방어적 프로그래밍, 테스트 가능성)을 이상적 구조를 향한 단계적 리팩토링으로 옮긴 것이다. 근거는 0818679 시점 코드베이스에 대한 5슬라이스 read-only 감사(약 48건)다. 디렉토리 Working Notes 지침에 따라 코드에 대한 사실(구체적 file:line 발견)과 후보 설계(목표 형태·단계 로드맵)를 분리해 기술한다.

1. 목표 & 북극성 목표 구조

xmux는 이미 두 축(axis) 위에 깔끔한 orthogonal 설계를 갖고 있다. 목표는 재작성이 아니다. 두 축을 순수 가산(purely additive)으로 확장 가능하게 만들고, 세 god-module을 분해하며, "머신"을 두 번째로 모델링해 중복을 살려두는 레거시를 걷어내는 것이다. 두 축의 확장은 확정돼 있으므로(WSL 머신 패밀리, 추가 mux 패밀리), 그 trait 추상화는 요구사항이며 YAGNI가 적용되지 않는다.

북극성 수용 기준.
  • 머신 패밀리(WSL) 추가 = 새 src/machine/wsl.rs + machine::wsl() 팩토리 + 단일 생성 사이트의 arm 한 줄. machine-kind에 대한 어떤 match/if 수정도 0, mux/ 내부 수정도 0.
  • tmux 호환 mux 패밀리 추가 = 새 src/mux/<kind>/ 디렉토리 + known_muxes() 엔트리 한 줄. 해당 패밀리는 identity + attach_plan + driver + enumerate + control_argv(+ 선택적 switch_in_place)만 구현하고, 공유 command-plan 동사는 공짜. host/·app/·state/·ui/ 수정 0.

2. 확정된 설계 결정

축 형태에 관한 두 질문을 계획 수립 전에 해소했고, 각각 독립적 2차 리뷰어(Codex)로 교차 검증했다. 비용은 배제하고 이상적 구조만 따졌다.

MACHINE 팩토리 — 결정: Option A (함수형 팩토리 + 단일 디스패치)

타입 있는 자유함수 팩토리(machine::local(socket), ssh(alias, control_path, os), 향후 wsl(distro))를 유지하고, 선택은 정확히 한 곳(Hosts::build)의 match src.kind에서만 수행한다. 세 번째 패밀리를 표현하지 못하는 Source.remote: bool은 MachineKind enum으로 교체한다. 기각: mux를 흉내낸 known_machines() 레지스트리.

원칙: 여기서 축 간 레지스트리 대칭은 피상적이다. mux 레지스트리는 런타임 문자열 탐지를 균일한 fn(String) 생성자로 지원하려고 존재한다. machine 축은 탐지가 없고(종류가 config 구조에서 확정) 생성자 입력이 이질적이다. fn(&Source) 레지스트리는 모든 팩토리가 Source 레코드 통짜를 삼키게 만들어 각 transport의 실제 의존성을 숨긴다. 구조는 장식적 대칭이 아니라 실제 force를 따른다.

MUX trait — 결정: 단일 trait 슬림화, 분할 안 함

항목결정
바이트 동일한 9개 command-plan 메서드(*_window_plan, *_session_plan, list_panes_plan)self.bin() 위의 trait 기본 구현. 호환성은 기본 동작이지 헬퍼 계층이나 TmuxCompatibleMux 베이스 trait이 아니다.
Mux를 역할 sub-trait로 분할(ISP)?안 함. trait은 항상 하나의 구체 타입으로 통째 구현되고 단일 확장점으로 소비된다. 분할은 하나의 계약을 흩뿌릴 뿐 독립 소비자를 만들지 못한다. 정직한 ISP 선: 서로 다른 소비자가 독립적으로 mock/구현할 역할이 있을 때만 분할 — 여기는 아니다. (감사의 "선택적 역할 분할"을 잘라냄.)
3개 옵션 in-place-switch 메서드(switch_client_argv / display_tty_record_prefix / switch_via_recorded_tty_cmd)하나로 통합: switch_in_place(host_key, session, display_tty) -> Option<SwitchPlan>가 드라이버가 blind 실행하는 불투명 계획을 반환. tmux의 tty >file 메커니즘이 mux 경계로 새는 것을 차단.
attach_plan의 window: Option<i64>(두 impl에서 미사용)제거. 현재 force 없음; 윈도우는 select_window_plan으로 선택.
동시성 제약 — 타 세션 존중. 다른 세션이 src/driver.rs, src/mux/tmux/display.rs, src/mux/psmux/display.rs를 편집 중이다(테스트 전용 Local::boxed/Ssh::boxed 생성자 마이그레이션). 이 브랜치는 깨끗한 HEAD에서 분기했고 그 작업트리 변경을 건드리지 않는다. 이 세 파일을 건드리는 작업(아래 HOT 표시)은 뒤(P5–P6)로 배치하거나 조율하며, 그 마이그레이션이 병합된 뒤 rebase한다. 프로덕션 영역과 테스트 영역이 달라 충돌은 작지만 실재한다.

3. 의도적으로 건드리지 않는 것

감사의 긍정 발견은 과잉 리팩토링(=rubric 자체의 KISS/YAGNI)에 대한 방어선이다. 아래는 건전하며 범위 밖이다:

  • mux 팩토리(known_muxes() + for_binary/for_kind/detect_backend): 중앙 kind-match 없음; mux 추가는 이미 가산적. 북극성의 모범 사례.
  • State::apply/apply_event: 순수 fold로 검증됨 — clock/registry를 데이터로 주입, I/O 없음. 수렴 원칙이 네비게이션/라이프사이클 action 집합에는 이미 성립.
  • machine/ 축 자체: Transport trait + Box<dyn Transport> 블랭킷 impl은 올바른 형태, 과잉추상화 아님.
  • display/ 계층: PTY/grid 메커니즘은 잘 분리돼 있고 실제 PTY 없이 테스트 가능(PtySink 뒤). 국소 수정 3건뿐, 재-레이어링 없음.
  • control 소켓 seam: 시맨틱 Action vs raw: 경계가 파싱단에서 깨끗하게 강제됨. 방어적 공백(크기 상한, enqueue 확인)만.
  • attach/session/vocab 모듈: 응집적이고 테스트 잘 됨.

6. 교차 불변식 (모든 단계의 리뷰 게이트)

  • 확장 축의 LSP 계약. 모든 Mux/Transport impl은 동일한 행동 계약을 지켜야 한다(메서드별 문서화). "고칠" 게 아니라 보존할 뉘앙스: 로컬 psmux enumerate는 Err를 반환하지 않는다(레지스트리가 권위 있음) — 의도된 per-transport 보장으로 문서화.
  • ISP 정직 선. 실제 소비자가 독립적으로 mock/구현할 부분집합이 필요할 때만 trait을 분할한다. 하나로 구현·소비되는 큰 trait은 단일 trait(기본 구현 포함)로 둔다.
  • Supervisor 무지(blindness). app/·ui/·state/는 mux/machine-특정 분기 0 — 동작은 driver_for(host) / host.mux / host.transport를 통해 흐른다.
  • 의존 방향. 도메인(model)은 오케스트레이션(app)에서 import하지 않는다. Selection을 아래로 내리는 것이 구체적 수정.
  • 경계에서 Fail-Fast. 시작 시 config 검증; 모든 외부 I/O 읽기(PTY, control 소켓)에 상한; 의미 있는 값을 조용히 기본값 처리 금지.
  • AS-IS. 주석/문서는 현재 불변식을 기술하며 변경 이력을 서술하지 않는다; 변경 기록은 git이 담당.

7. 순서 & HOT 파일 조율

  • HOT 표시 행은 driver.rs / mux/tmux/display.rs / mux/psmux/display.rs(타 세션 파일)를 건드린다. P5–P6에 집중(단, S4-M1의 display.rs 호출부는 P1에서 조율). 그 마이그레이션이 병합된 뒤 rebase; 프로덕션 vs 테스트 영역이 달라 작은 충돌만 예상.
  • P0과 P4는 레지스트리 작업과 독립이라 앞쪽/병렬로 진행 가능.
  • P1이 사용자 우선순위인 확장성을 전달하고 P2의 transport 사이트 완전 제거를 열어준다.
  • 모든 단계는 완료 게이트에서 끝난다: cargo test + clippy + fmt 클린 + 단계별 검증; 구조 단계(P2·P4·P5)는 추가로 사람 라이브 게이트 필요.

8. 산출물 범위

이 문서는 마스터 로드맵이다. 다음 단계는 writing-plans로 P0를 상세화하는 것(실행 가능한 테스트 우선 구현 계획). 이후 단계는 각자 spec→plan 사이클을 밟아, 결정이 직전 단계가 남긴 상태에 근거하도록 한다. 어떤 단계든 실행은 별도의 승인 단계이며, 이 스펙은 어떤 코드 변경도 약속하지 않는다.
